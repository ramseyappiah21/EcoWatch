CHAPTER FOUR

SYSTEM IMPLEMENTATION

4.1 Introduction

This chapter presents the implementation of EcoWatch Tarkwa, a multi-channel civic environmental reporting platform for the Tarkwa-Nsuaem Municipal Assembly (TNMA). The delivered system comprises a Flutter citizen application, a Node.js/Express REST API with PostgreSQL, a web-based admin portal for agency users, and a USSD channel via Africa’s Talking. The chapter focuses on what was built and how the main modules work, with screenshots of the running application. Design rationale is covered in Chapter Three and is not repeated here except where it clarifies an implementation choice.

4.2 Development Environment

Development used Flutter/Dart for the cross-platform client, Node.js with Express for the API, and PostgreSQL for persistence. Supporting tools included Docker Compose for the database, Dio for HTTP, flutter_map and Geolocator for mapping and GPS, SharedPreferences for local offline storage, and Africa’s Talking for USSD (and optional SMS). The admin portal is a static web application served by the same backend. Citizen use is anonymous; agency access uses JWT authentication against role-seeded accounts.

4.3 Citizen Mobile Application

The citizen application is anonymous: users do not create accounts. Navigation uses a bottom shell with Home, Track, Maps, Alerts, and Profile, plus a floating action button that opens Report Incident. Localisation supports English and Ghanaian language options configured in the app. Screens that exist in the codebase for in-app staff login/dashboard are not part of the citizen navigation; officials use the web admin portal instead.

4.3.1 Home screen

The home screen summarises recent local activity and provides quick entry to reporting and tracking. Reports cached on the device are shown first so the interface remains usable on slow networks; a background refresh updates status from the server when connectivity is available.

4.3.2 Report Incident

Reporting is the primary citizen workflow. The user selects an environmental category, writes a short description, optionally attaches a photograph, confirms or adjusts the incident location on a map, and submits. On successful upload the server returns a tracking token in the form EW-…, which the citizen can use later without revealing personal identity.

Location uses device GPS where permitted. The map shows the user’s approximate position and a distinct incident pin that can be moved before submission. If GPS is unavailable, the user can still place the pin manually. Offline, the report is stored locally with a pending-upload status and is synchronised when the network returns.

4.3.5 Offline behaviour

Reports are written to local storage first. When the device is online, the client attempts immediate upload; otherwise the report remains queued (pendingUpload). An automatic sync service retries pending items when connectivity is restored. This design matches intermittent mobile data conditions common in the field.

4.4 Backend API and Database

The Express API exposes versioned routes under /v1 for public reporting and tracking, authentication for officials, maps and analytics data, sync, and admin operations. Media uploads are handled on the server (local storage in the current deployment). PostgreSQL stores users (agency roles), categories, reports, status history, hotspots, announcements, and audit-related records as defined in the project schema and seed scripts.

Incident categories are linked to responsible agencies so that new reports can be routed for multi-agency handling. Citizen endpoints do not require login; admin endpoints require a valid JWT and enforce role-based access (for example municipal admin, agency admin, environmental officer, researcher, and super admin).

4.5 Admin Portal

Municipal and agency staff use the web admin portal (served at /admin), not the citizen app. Access is JWT-based. Tabs and actions are shown according to role so that a field officer, agency administrator, municipal administrator, researcher, or super administrator each sees an appropriate workspace. Demo accounts were seeded for demonstration (for example municipal@ecowatch.gov, agency accounts such as epa@ecowatch.gov, and officer accounts such as epa.officer@ecowatch.gov).

There is one shared official portal at /admin. After login, the interface changes by role (header title, Reports tab label, visible tabs, and row actions). That is how the municipal desk, agency desk, field-officer workspace, emergency desk, police-support desk, and research view are delivered—without separate mobile “officer apps.”

4.6 USSD Channel

USSD provides access for users without smartphones or reliable app installation. The sandbox short code used during implementation and testing is *384*63693# (Africa’s Talking). The backend webhook (POST /v1/ussd/webhook) drives a session menu to report an incident by category, track by token, and show brief privacy/help text. USSD reports are stored like app reports but with simplified location handling suitable for keypad input. Production deployment would use an approved live short code and a publicly reachable HTTPS callback URL.

4.7 Hotspot Detection and Prediction

Operational hotspot detection clusters recent geolocated reports using DBSCAN in the backend service layer. The configuration used in implementation treats reports within approximately 1 km (eps) as neighbours and requires a minimum cluster size of 5 points (minPts, with adaptive lowering when data are sparse). A scheduled job recomputes hotspots periodically (on the order of every six hours), persists results, and exposes them for maps and analytics. Priority is derived from report density and related attributes so officers can focus on areas of repeated concern. These outputs are what the portal Map and Analytics tabs display (Figures 4.11 and 4.12).

Complementary predictive work lives under the project’s ML hotspot module: training and prediction scripts can export prediction artefacts that the API may serve when available; otherwise the service falls back to heuristic scoring. DBSCAN is the primary operational detector; ML prediction supports anticipatory planning.

4.8 Integration Testing and Deployment Notes

Implementation was verified by running the API against PostgreSQL (Docker), exercising citizen flows on Android (release APK) and on mobile browsers via Flutter web where HTTPS was required for location (for example iPhone Safari through a tunnelled public URL). Africa’s Talking sandbox dialling confirmed USSD menu behaviour when the webhook was publicly reachable. Remaining limitations that affect deployment—such as cloud object storage for media, a live USSD short code, and production hosting—are summarised in Chapter Five rather than expanded here.

4.9 Chapter Summary

EcoWatch Tarkwa was implemented as an integrated stack: an anonymous Flutter client for reporting and tracking, a secured Express/PostgreSQL backend, a role-based admin portal, USSD access for feature-phone users, and DBSCAN-based hotspot detection with optional predictive outputs. The following chapter concludes the project and outlines limitations and future work.

CHAPTER FIVE

CONCLUSION AND RECOMMENDATIONS

5.1 Introduction

This chapter summarises the achievements of the EcoWatch Tarkwa project, states its limitations honestly, and recommends future work. It closes the dissertation by relating the implemented system to the original aim of supporting anonymous, multi-channel environmental incident reporting for the Tarkwa-Nsuaem area.

5.2 Summary of Achievements

The project delivered a working civic reporting platform with the following outcomes:

  • Anonymous mobile reporting with category selection, media, map-based location, severity, and tracking tokens.
  • Offline-first submission with local queueing and automatic synchronisation when connectivity returns.
  • Status tracking for citizens using tokens, without requiring user accounts.
  • Multi-agency backend and admin portal for case handling, status updates, announcements, and analytics.
  • USSD channel integrated with Africa’s Talking for users without full smartphone app access.
  • Hotspot detection using DBSCAN on recent reports, with optional ML-based prediction artefacts for planning support.

Together these components demonstrate a practical architecture for TNMA and partner agencies to receive and act on community environmental reports.

5.3 Limitations

  • Hosting and scale: The demonstration environment used local/LAN deployment and tunnels for HTTPS; production would need managed cloud hosting, backups, and monitoring.
  • Media storage: Uploads are stored on the application server rather than durable object storage (for example S3-compatible storage).
  • USSD: Testing used the Africa’s Talking sandbox short code; a live short code and always-on public webhook are required for public rollout.
  • On-device image AI: Photo category suggestion in the client is not a production trained vision model; category selection remains primarily user-driven.
  • USSD geolocation: Keypad reporting uses simplified location defaults compared with GPS-assisted app reports.
  • Evaluation scope: Formal large-scale field trials with TNMA officers and residents were outside the core build scope of this academic project.

5.4 Recommendations and Future Work

  1. Deploy the API, database, and admin portal to a production cloud environment with HTTPS, backups, and role review with TNMA.
  2. Migrate media to object storage and add retention policies aligned with privacy requirements.
  3. Obtain a production USSD short code and complete Africa’s Talking live integration and SMS notifications.
  4. Train and validate a real on-device or server-side image classifier for the project’s incident categories.
  5. Conduct a pilot with municipal and agency officers, collect usability feedback, and refine workflows and hotspot parameters for Tarkwa’s geography.
  6. Strengthen privacy documentation and consent messaging for public launch.

5.5 Conclusion

EcoWatch Tarkwa shows that anonymous, multi-channel environmental reporting is feasible for a Ghanaian municipal context by combining a Flutter citizen app, a structured backend with agency routing, a practical admin portal, USSD reach, and spatial hotspot analysis. Despite limitations typical of a final-year prototype—especially around production hosting, live USSD, and field evaluation—the system meets its core objective: giving residents a safer, clearer path to report environmental incidents and giving authorities a consolidated way to receive, track, and prioritise them. With the recommendations above, the platform can move from academic demonstration toward operational use in the Tarkwa-Nsuaem Municipal Assembly.
