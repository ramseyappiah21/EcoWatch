CHAPTER FOUR

SYSTEM IMPLEMENTATION

4.1 Introduction

This chapter presents the implementation of EcoWatch Tarkwa, a multi-channel civic environmental reporting platform for the Tarkwa-Nsuaem Municipal Assembly (TNMA). The delivered system comprises a Flutter citizen application, a Node.js/Express REST API with PostgreSQL, a web-based official portal for municipal and agency users, and a USSD channel via Africa's Talking. The chapter focuses on what was built and how the main modules work, with screenshots of the running application. Design rationale is covered in Chapter Three and is not repeated here except where it clarifies an implementation choice.

4.2 Development Environment

Development used Flutter/Dart for the cross-platform client, Node.js with Express for the API, and PostgreSQL for persistence. Supporting tools included Docker Compose for the local database, Dio for HTTP, flutter_map and Geolocator for mapping and GPS, SharedPreferences for offline storage, and Africa's Talking for USSD (and optional SMS). The official portal is served under /admin. Citizen use is anonymous; officials authenticate with JWT. The project is on GitHub and was deployed to Render with managed PostgreSQL for HTTPS access.

4.3 Citizen Mobile Application

The citizen application is anonymous. On first launch the user sees a splash (start) screen, then a short onboarding sequence, then the main shell: Home, Track, Maps, Alerts, and Profile, plus a floating button for Report Incident. Officials use the web portal, not in-app staff screens.

4.3.1 Splash (starting) screen

When the app opens, a splash screen branded with the EcoWatch icon and name appears briefly on the primary green background. It checks whether the user has already completed onboarding in this install/session. If onboarding is finished, the app proceeds to Home; otherwise it opens the onboarding flow.

4.3.2 Onboarding screens

Onboarding introduces the app in a short swipeable sequence before the citizen reaches Home. The pages explain, in order: welcome to EcoWatch Tarkwa; how to report an incident; how to track a report with a token; how maps support awareness; and that USSD (*384*63693#) is available for users without the full app. The user can move Next through the pages and finish with Get Started. After completion, onboarding is not shown again until the intro flag is cleared.

4.4 Backend API and Database

The Express API under /v1 covers public reporting/tracking, official authentication, maps, analytics, sync, and admin operations. PostgreSQL stores users, roles, reports, hotspots, announcements, notifications, and audit records. Categories drive multi-agency routing. Portal endpoints require JWT and role checks.

4.5 Official Admin Portal

All officials use one portal at /admin. After login, EcoWatch shows only the tabs allowed for that person's role. The subsections below explain (1) what each tab means, (2) which tabs each role sees, then (3) each workspace with screenshots. Demo passwords are documented separately and are not shown on the login page.

How to read the system: administrators get management tabs (Performance, Users; municipal/super also Announcements; super also Audit and Settings). Field officers, emergency officers, and police see only four tabs - Reports, Map, Analytics, Notifications. Researchers also see those four, but Reports is anonymised.

4.5.3 Same layout for every agency (generalised view)

A reader does not need a separate screenshot for EPA, WRC, NADMO, Fire, Forestry, Waste, and Police. All participating agencies use the same portal screens and the same tab names. The system is generalised by role, not redesigned per agency.

Agency administrator (any agency) always sees these six tabs after login:

  • Agency Incidents (Reports)
  • Map
  • Analytics
  • Notifications
  • Performance
  • Agencies & Users

What changes between agencies (so one EPA screenshot represents WRC, NADMO, Fire, and the rest):

  • Header text - shows that agency’s name (for example “Environmental Protection Agency” vs “NADMO”).
  • Which incidents appear - Reports, Map, and Analytics only list cases routed to that agency’s mandate.
  • Which officers can be assigned - the Assign list is limited to that agency’s staff.

Agency field officer (any agency) always sees the same four tabs: My Cases, Map, Analytics, Notifications. Only the header and the assigned cases change (EPA officer vs Forestry officer, and so on).

4.7 Hotspot Detection

DBSCAN clusters recent reports (eps about 1 km, minPts = 5) to surface areas with concentrated incidents. Results appear on the portal Map and Analytics tabs, where a 30-day rolling hotspot-growth chart shows how clustering changes over time.

4.8 Testing and Deployment

Verified with local Docker PostgreSQL, Android APK, Flutter web over HTTPS where needed, and Africa's Talking sandbox USSD. The API and portal were deployed to Render (Docker + managed Postgres), with migration on boot and optional seed-on-boot for demo accounts.

4.9 Chapter Summary

EcoWatch Tarkwa implements an anonymous citizen app, Express/PostgreSQL backend, a role-based portal whose tabs depend on who logs in (Tables 4.2–4.4), USSD access, hotspot analytics, and cloud hosting. Chapter Five concludes the project.

CHAPTER FIVE

CONCLUSION AND RECOMMENDATIONS

5.1 Introduction

This chapter summarises achievements, limitations, and recommendations for EcoWatch Tarkwa.

5.2 Summary of Achievements

  • Anonymous mobile reporting with media, map location, severity, and tracking tokens.
  • Offline-first submission and token-based tracking without citizen accounts.
  • Role-based official portal with clear tabs per role; agencies share one generalised layout (same tabs, different scope).
  • Splash and onboarding for first-time citizen users, then Home / Track / Maps / Alerts / Profile.
  • USSD channel via Africa's Talking.
  • DBSCAN hotspot detection with a 30-day hotspot-growth trend.
  • Cloud-hosted API and portal with a professional login screen.

5.3 Limitations

  • Free-tier cloud hosting may sleep when idle.
  • Media stored on the app server, not object storage.
  • USSD sandbox only; live short code and SMS still needed for full rollout.
  • Image AI is not a production vision model.
  • Large-scale field trials with TNMA were outside this academic build.

5.4 Recommendations and Future Work

  1. Use stable always-on hosting with backups.
  2. Move media to object storage with retention policies.
  3. Obtain production USSD and SMS.
  4. Train a real image classifier for categories.
  5. Pilot with TNMA and agencies; refine hotspot parameters.
  6. Replace demo accounts before public launch.

5.5 Conclusion

EcoWatch Tarkwa shows that anonymous, multi-channel environmental reporting is feasible for Tarkwa-Nsuaem by combining a Flutter citizen app, a multi-agency backend, a portal whose tabs make each official's job clear, USSD reach, hotspot analysis, and HTTPS deployment. With the recommendations above, the platform can move from academic demonstration toward operational municipal use.
